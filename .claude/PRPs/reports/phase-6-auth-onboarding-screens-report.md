## Deviations from Plan
- **`computeNextPaymentDate` timezone bug**: the plan's own `IMPLEMENT` code built the candidate date with `new Date(year, month, day)` (local midnight) then serialized with `.toISOString().slice(0, 10)`. `toISOString()` converts to UTC, so in any timezone ahead of UTC (this environment is UTC+3) local midnight shifts to the *previous* day in the UTC string — verified via Node REPL: `computeNextPaymentDate(31, new Date('2026-02-15'))` returned `2026-02-27` instead of the plan's own expected `2026-02-28`. Fixed by formatting the date string directly from the `Date`'s local `getFullYear`/`getMonth`/`getDate` components instead of routing through UTC. Also tightened the "has this month's day passed" comparison to be date-only (not date+time), so a `payDay` equal to today's date-of-month resolves to *this* occurrence rather than always rolling to next month. Re-verified against the plan's exact test cases — both now match.

## Issues Encountered
- Expo Router's typed routes (`app.json` → `experiments.typedRoutes: true`) require `.expo/types/router.d.ts` to be regenerated by the Metro dev server before `tsc` will accept the new `(auth)`/`(onboarding)` route strings. Resolved by running `npx expo start --no-dev` briefly (killed after ~20s) to trigger regeneration, then re-running `typecheck` — matches the plan's own Task 8 GOTCHA note.
